- 데이터베이스 구조: [`./database.md`](./database.md)
- 보안 감사: [`./security_audit.md`](./security_audit.md)

## 1) 개요

현재 구조는 소규모 트래픽에서 안정적으로 동작하나, 인증이 필요한 요청에서 DB 조회가 누적되며 네트워크 지연이 응답 시간에 직접 반영됩니다.

## 2) 주요 관찰 지표

| 구분 | 관찰값(문서 기준) | 비고 |
| :--- | :---: | :--- |
| 단일 DB 쿼리 (`SELECT 1`) | 약 33ms | 네트워크 왕복 시간 영향 |
| 권한/사용자 조회 누적 | 약 80~100ms | 보호 라우트 기준 |
| 초기화(첫 연결+첫 쿼리) | 약 360ms | 콜드 스타트 체감 |

> 위 수치는 기존 감사 기준값이며, 환경/DB 위치에 따라 달라질 수 있습니다.

## 3) 상세 분석

### 3.1 인증/권한 경로 오버헤드

- `get_active_user` 의존성은 사용자 조회 + 필수 약관 동의 확인을 수행합니다.
- 보호 라우트에선 요청마다 DB 왕복이 발생합니다.

### 3.2 데이터베이스 레이어

- 주요 조회 컬럼 인덱스는 이미 존재합니다(`users.email`, `daily_snippets.user_id/date` 등).
- SQLAlchemy async 엔진/세션 풀링을 사용합니다.

## 4) 권장 개선안

### 단기

1. **읽기 캐시 적용 범위 정의**
   - 자주 변하지 않는 메타성 데이터(예: 일부 권한/정책)를 캐시 후보로 분리합니다.
2. **핫패스 쿼리 축소**
   - 라우트별로 실제 필요한 데이터만 조회하도록 점검합니다.
3. **측정 자동화**
   - `pytest` 또는 스크립트 기반으로 핵심 API 응답 시간 스냅샷을 정기 수집합니다.

### 중장기

1. **분산 캐시(예: Redis) 도입 검토**
   - 다중 인스턴스 환경에서 캐시 일관성을 확보합니다.
2. **쿼리 합치기/사전로딩 최적화**
   - 동시 조회되는 리소스를 JOIN/적절한 로딩 전략으로 줄입니다.

## 5) 운영 체크리스트

- [ ] 보호 라우트 평균 응답시간(백분위) 모니터링
- [ ] DB 커넥션 풀 사용량/대기시간 모니터링
- [ ] 배치(`run_daily_achievement_grants.py`) 실행 시간 추적

## 6) 결론

현재 성능 병목은 **네트워크 + 인증/권한 경로의 누적 조회 비용**에 집중되어 있습니다.
단기적으로는 핫패스 축소와 측정 자동화, 중장기적으로는 캐시 전략 도입이 효과적입니다.
