# 성능 감사 보고서 (Performance Audit Report)

이 보고서는 `gcs-auth` 서비스의 현재 아키텍처와 데이터베이스 상호작용을 기반으로 한 성능 분석 결과입니다.

## 1. 개요 (Executive Summary)
현재 시스템은 소규모 환경에서 안정적으로 작동하지만, 모든 보호된 요청마다 발생하는 다중 데이터베이스 조회 작업이 주요 성능 병목 구간으로 식별되었습니다. 네트워크 지연(Latency)이 전체 마이크로서비스 응답 시간에 직접적인 영향을 미치고 있습니다.

## 2. 주요 측정 지표 (Key Metrics)

| 구분 | 평균 소요 시간 (Latency) | 비고 |
| :--- | :---: | :--- |
| **단일 DB 쿼리 (SELECT 1)** | ~33ms | Supabase 네트워크 왕복 시간 포함 |
| **권한 정보 조회** | ~45ms | `route_permissions` 테이블 조회 |
| **미들웨어 합계 지연** | **~80-100ms** | 권한 조회 + 사용자 조회 (보호된 라우트 기준) |
| **전체 유닛 초기화** | ~360ms | DB 연결 및 첫 쿼리 실행 합계 |

## 3. 상세 분석 (Detailed Analysis)

### 3.1 미들웨어 오버헤드 (Middleware Overhead)
`check_route_permissions` 의존성이 글로벌하게 적용되어 있어 다음 이슈가 발생합니다:
- **매 요청당 DB 호출**: 모든 요청 시 `route_permissions`를 조회합니다.
- **보호된 라우트의 추가 지연**: 인증이 필요한 라우트의 경우 `users` 테이블을 추가로 조회하여 최신 역할을 확인합니다.
- **네트워크 종속성**: 외부 DB(Supabase)를 사용하므로 네트워크 상태에 따라 응답 속도가 크게 출렁일 수 있습니다.

### 3.2 데이터베이스 구조
- **인덱스 최적화**: `users(google_sub)`, `route_permissions(path, method)` 등 주요 검색 조건에 대한 인덱스가 적절히 생성되어 있어 쿼리 실행계획 자체는 효율적입니다.
- **연결 풀링**: SQLAlchemy의 비동기 엔진을 통해 커넥션 풀링을 사용하고 있어 연결 생성 비용은 최소화되어 있습니다.

## 4. 권장 최적화 방안 (Recommendations)

### 🚀 단기 과제 (Short-term)
1. **권한 설정 캐싱 (Permission Caching)**:
   - `route_permissions` 데이터는 자주 변경되지 않으므로, 애플리케이션 시작 시 메모리에 로드하거나 간단한 인메모리 캐시(예: Python `dict` 또는 `lru_cache`)를 적용하여 DB 호출을 제거합니다. (예상 성능 향상: ~40ms)
2. **세션 기반 역할 활용**:
   - 매번 DB에서 유저를 조회하는 대신, 세션에 저장된 역할을 사용하되 필요 시에만(예: 역할 변경 직후) 갱신하는 로직을 검토합니다. (예상 성능 향상: ~40ms)

### 📈 장기 과제 (Long-term)
1. **Redis 도입**: 분산 환경에서의 캐싱 및 속도 제한(Rate Limiting) 상태 관리를 위해 Redis를 도입하여 안정성을 높입니다.
2. **Batching/Prefetch**: 동의 여부 확인(`get_active_user`) 시 사용자 정보와 약관 정보를 한 번의 쿼리(JOIN)로 가져오도록 최적화합니다.

## 5. 결론
현재 서비스의 평균 응답 오버헤드는 **약 100ms** 수준으로 감지되었습니다. 위 권장 사항(특히 캐싱)을 적용할 경우, 보호된 라우트의 응답 속도를 **최대 80% 이상(20ms 미만)**으로 단축할 수 있을 것으로 판단됩니다.
